# จากไอเดียสู่ระบบจริง: กรณีศึกษาสร้างระบบ QR Check-in ด้วย AI

---

## บทนำ

เอกสารนี้บันทึกกระบวนการสร้างระบบเช็คชื่อนักเรียนด้วย QR Code ตั้งแต่ต้นจนใช้งานได้จริง โดยใช้ AI เป็นผู้ช่วยหลักตลอดกระบวนการ

สิ่งที่น่าสนใจคือผู้พัฒนาไม่ได้แค่สั่งงาน AI แต่มีการ **โต้ตอบ เสนอไอเดีย ตั้งคำถาม และปรับทิศทางร่วมกัน** ตลอดเวลา ซึ่งเป็นลักษณะการทำงานกับ AI ที่ได้ผลดีที่สุด

> **วิธีอ่านเอกสารนี้**
> - 💬 `ผู้พัฒนา` — prompt หรือคำถามที่ใช้จริง
> - 🤖 `AI` — สรุปสิ่งที่ AI ตอบหรือเสนอ
> - 💡 `บทเรียน` — สิ่งที่ได้จากการปรึกษาครั้งนั้น

---

## Phase 1: สำรวจไอเดียและเลือกแนวทาง

### บริบท
ผู้พัฒนามีโจทย์ชัดเจนว่าต้องการระบบเช็คชื่อที่ป้องกันการเช็คชื่อแทนกัน แต่ยังไม่แน่ใจว่าควรใช้เทคโนโลยีอะไร จึงเริ่มต้นด้วยการถามแบบกว้างๆ ก่อน

---

💬 **ผู้พัฒนา**
```
ผมอยากทำระบบเช็คชื่อนักเรียนในห้อง
โดยให้เฉพาะคนที่เข้าห้องเรียนจริงๆ เช็คอินได้เท่านั้น
มีไอเดียอะไรบ้าง?
```

🤖 **AI เสนอ 6 แนวทาง**
- Dynamic QR Code — เปลี่ยนทุก N วินาที
- WiFi Fingerprint — ตรวจ network ที่เชื่อมต่อ
- GPS Geofencing — กำหนดพิกัดรอบห้อง
- Bluetooth Beacon — ตรวจ signal ในห้อง
- OTP จากอาจารย์ — รหัสสั้นอายุสั้น
- Hybrid QR + WiFi — รวมสองวิธี

พร้อมเปรียบเทียบความยาก และระดับความปลอดภัยของแต่ละวิธี

---

💬 **ผู้พัฒนา**
```
อธิบายวิธีทำระบบ check-in ด้วย QR Code
ที่เปลี่ยนตามเวลาให้ละเอียด
```

🤖 **AI อธิบาย**
- หัวใจของระบบคือ Time-based Token
- Server สร้าง token ใหม่ทุก N วินาที ผูกกับ `class_id + timestamp`
- QR ที่หมดอายุจะใช้ไม่ได้อัตโนมัติ
- อธิบาย architecture ทั้ง frontend, backend, database

---

💡 **บทเรียน**
การเริ่มต้นด้วยคำถามกว้างๆ ช่วยให้เห็นภาพรวมของทางเลือกทั้งหมดก่อน แทนที่จะ commit กับแนวทางแรกที่นึกได้ทันที

## Phase 2: ออกแบบ Security

### บริบท
หลังจากเลือก Dynamic QR แล้ว ผู้พัฒนาเริ่มคิดถึงช่องโหว่ที่อาจเกิดขึ้น และตั้งคำถามที่คมคายขึ้น

---

💬 **ผู้พัฒนา**
```
ถ้า 90 วิ ก็นานพอที่จะถ่ายภาพ
แล้วส่งต่อให้คนข้างนอกสแกนได้ใช่ปะ?
```

🤖 **AI ยอมรับและเสนอทางแก้**
นั่นคือจุดอ่อนหลักของ QR อย่างเดียว แนะนำสองแนวทาง คือ WiFi fingerprint และ Face liveness แต่ทั้งสองมีข้อจำกัด

---

💬 **ผู้พัฒนาเสนอไอเดียใหม่**
```
ผมมีไอเดีย QR เปลี่ยนทุก 5 วิ และบันทึกเวลาที่สแกน
กับเวลาที่กดส่ง ถ้า pattern ปกติก็ valid
ถ้าส่งไลน์ไปก็ไม่ทันแน่นอน คิดว่าไง?
```

🤖 **AI วิเคราะห์**
ไอเดียนี้ใช้ได้จริง แต่มีจุดที่ต้องระวังคือ timestamp จาก client อาจถูก tamper ได้ ควรให้ server จับเวลาเองแทน

---

💬 **ผู้พัฒนาปรับ approach**
```
เราไม่ต้องถึงกับเข็ม เช็คได้ 100%
แต่ถ้าแบบคนทั่วไปเช็คได้ก็โอเค
ระบบจะได้ไม่ยุ่งยากไป
ผมว่าไม่ reject แต่มีระบบหลังบ้านตรวจสอบ
ความผิดปกติแทน นร. จะได้ไม่รู้ว่าระบบเช็คได้
คิดว่าไง?
```

🤖 **AI เห็นด้วยและขยายความ**
นี่คือ approach แบบเดียวกับที่บริษัทใหญ่ๆ ใช้จริง เก็บ signal ไว้แล้วค่อย flag ภายหลัง แนะนำ signal ที่ควรเก็บเพิ่มเติม เช่น device fingerprint, scan gap, submission pattern

---

💡 **บทเรียน**
AI ไม่ได้แค่รับคำสั่ง แต่ช่วย validate ไอเดียและชี้จุดที่อาจมีปัญหา ขณะเดียวกันผู้พัฒนาก็เสนอ approach ที่มาจาก domain knowledge จริงๆ ซึ่ง AI ยืนยันว่าถูกต้อง

---

## Phase 3: ออกแบบ Payload ใน QR

### บริบท
เมื่อทิศทางชัดแล้ว ต้องออกแบบว่าจะเก็บข้อมูลอะไรใน QR และมากแค่ไหน

---

💬 **ผู้พัฒนา**
```
มาคุยเรื่องข้อมูลที่เราจะใส่ใน QR หน่อย
ว่าควรมีอะไรบ้าง ที่เราจะ refresh ทุก 5 วิ
```

🤖 **AI เสนอโครงสร้าง**
```json
{
  "sid": "CS101-20250613-6849a2f1",
  "t": 1749823400,
  "seq": 12,
  "sig": "a3f9c2"
}
```
อธิบายหน้าที่ของแต่ละ field

---

💬 **ผู้พัฒนาตั้งคำถามเรื่อง QR density**
```
ถ้ายิ่งข้อมูลเยอะ QR ก็ต้องเก็บรายละเอียดเยอะ
ทำให้ QR รอยหยัก จะเล็กลง มีผลต่อการสแกนไหม?
```

🤖 **AI แนะนำให้ใช้ reference แทน**
อย่าใส่ข้อมูลจริงใน QR ใส่แค่ reference ไปหา server แทน
```
https://domain.com/c?s=a3f9c2
```
ข้อมูลจริงทั้งหมดอยู่ที่ server QR จึงสั้นและสแกนง่าย

---

💬 **ผู้พัฒนาคิดต่อไปอีก**
```
qr นี้แค่บรรจุ s t ได้ก็พอใช่มั๊ย?
แล้วทำไมเราไม่เอาแค่ s=a3f9c2ef&t=1749823400
ไม่ต้องมี URL เลย?
```

🤖 **AI ยืนยัน**
ถูกต้องเลย jsQR อ่านค่าออกมาแล้วไม่ได้เปิด URL จริงๆ ดังนั้น QR บรรจุแค่ `s` และ `t` ก็พอ QR จะสั้นที่สุด สแกนง่ายที่สุด

---

💡 **บทเรียน**
การตั้งคำถามว่า "จริงๆ ต้องการอะไรบ้าง?" ช่วยลด complexity ได้มาก วิวัฒนาการของ QR URL จาก JSON ซับซ้อนมาเป็น `s=xxx&t=xxx` เกิดจากการตั้งคำถามหลายรอบ

---

## Phase 7: แก้ปัญหา Elapsed คำนวณผิด

### บริบท
ทดสอบโดยแคปหน้าจอรอ 15 วิแล้วสแกน แต่ elapsed แสดงแค่ 2 วิ

---

💬 **ผู้พัฒนา**
```
ผมลองเทส แคปหน้าจอรอสัก 15 วิ
ทำไม ระยะเวลา ยังขึ้น แค่ 2 วิ อยู่
```

🤖 **AI วิเคราะห์ปัญหา**
สูตรเดิม `floor(t_receive / 3) * 3` ได้แค่ "ตำแหน่งใน window 3 วิ" เสมอ ได้สูงสุดแค่ 2 วิ ไม่ว่าจะรอนานแค่ไหนก็ตาม

---

💬 **ผู้พัฒนา**
```
ผมสงสัยว่า ทำไมเราไม่เอา sig มาหาเวลา
แล้วเทียบกับ t_receive ซึ่งก็มีอยู่แล้ว
```

🤖 **AI อธิบาย**
`sig` คือ HMAC ที่ hash แล้ว ย้อนกลับไม่ได้ จึงดึง `t` ออกมาจาก `sig` ไม่ได้ วิธีเดียวที่ทำได้คือใส่ `t` ลงใน QR โดยตรง

```
s=a3f9c2ef&t=1749823400
```

JavaScript ดึง `t` ออกมาแล้วคำนวณ

```javascript
elapsed = Math.floor(Date.now() / 1000) - t_qr
```

---

💬 **ผู้พัฒนาตั้งคำถามเพื่อความเข้าใจ**
```
t_qr = floor(t_scan / 3) * 3 ทำไมเป็นสูตรนี้ คืออย่างไร?
```

🤖 **AI อธิบายด้วยตัวอย่าง**
```
t_scan = 16 → floor(16/3)*3 = 15  ← ต้น window เดียวกับ t=15
t_scan = 17 → floor(17/3)*3 = 15  ← ต้น window เดียวกัน
t_scan = 18 → floor(18/3)*3 = 18  ← ต้น window ใหม่
```

ผู้พัฒนาตอบว่า "ได้ เข้าใจหมดแล้ว แก้เลย"

---

💡 **บทเรียน**
การถามว่า "ทำไมเป็นสูตรนี้?" แทนที่จะแค่รับโค้ดมาใช้ ช่วยให้เข้าใจ logic จริงๆ และสามารถ debug ได้เองในอนาคต


🤖 **AI อธิบายความต่างระหว่าง "รันได้" กับ "ถูกต้อง"**
บางอย่างรันได้แต่แอบซ่อน bug ไว้ เช่น explode sid จะพังเฉพาะกรณีที่ชื่อวิชามี `-` ซึ่งอาจไม่เจอในการทดสอบปกติ

---

💬 **ผู้พัฒนาเสนอวิธีแก้ที่ดีกว่า**
```
เอาเป็นตัด 10 ตัวท้ายได้มั๊ย วันที่เราบังคับใส่รูปแบบนี้อยู่แล้ว
ชื่อวิชา Dis-Math-th ก็ยังตัดได้
```

🤖 **AI เห็นด้วย**
ดีกว่าเลยครับ เพราะวันที่ `YYYY-MM-DD` มี 10 ตัวเสมอ

```php
$date    = substr($sid, -10);
$prefix  = substr($sid, 0, -11);
$dash    = strrpos($prefix, '-');
$course  = $dash !== false ? substr($prefix, 0, $dash) : $prefix;
$section = $dash !== false ? substr($prefix, $dash + 1) : '';
```

---

💬 **ผู้พัฒนาสังเกตเพิ่มเติม**
```
วันที่ที่เราตัดเอาไปใช้ทำอะไรนะ?
```

🤖 **AI ตรวจสอบแล้วพบว่า**
จริงๆ แล้ว `date` ส่งกลับไปใน response แต่ `checkin_form.php` ไม่ได้ใช้ `d.date` เลย สามารถตัดออกได้ทั้งหมด

💬 **ผู้พัฒนา**
```
ที่เราอุตส่าห์ตัดไปไม่ได้ใช้ นี่นะ
```

---

💡 **บทเรียน**
การ review โค้ดหลัง implement เสร็จสำคัญมาก และการที่ผู้พัฒนาตั้งคำถามว่า "ทำไมรันได้?" และ "ตัดไปทำอะไร?" ช่วยให้โค้ดสะอาดขึ้นอย่างมีเหตุผล ไม่ใช่แค่แก้ตามที่ AI บอก

💬 **ผู้พัฒนา**
```
Fatal error: Call to undefined function random_bytes()
อ้อ host จริงๆ ใช้ PHP v.5
```

🤖 **AI แนะนำ**
PHP 5 ไม่มี `random_bytes()` เปลี่ยนเป็น `openssl_random_pseudo_bytes()` แทน

---

💬 **ผู้พัฒนา**
```
ถ้าอยากให้คนอื่นเอาไปใช้ได้ ควรแก้ไงดี?
```

🤖 **AI เสนอให้แยก config**
สร้าง `config.php` เก็บ password และ settings ทั้งหมด แล้วใส่ใน `.gitignore` คนอื่นสร้างเองได้จาก README

💬 **ผู้พัฒนา**
```
หรือเค้าดูโค้ดอื่นก็น่าจะพอเดาได้ว่าควรมีอะไร
```

🤖 **AI เห็นด้วย**
ใช่ครับ ใน README อธิบายไว้ครบหมดแล้ว ไม่จำเป็นต้องมี `config.example.php` เพิ่ม

---

💬 **ผู้พัฒนา**
```
.DS_Store ตามไปด้วย ทำไงดี?
```

🤖 **AI แนะนำ**
```bash
git rm --cached .DS_Store
echo ".DS_Store" >> .gitignore
git add .gitignore
git commit -m "remove .DS_Store and add to gitignore"
git push
```

💡 **บทเรียน**

## สรุป: Pattern การทำงานกับ AI ที่ได้ผล

### AI ทำได้ดี
- เสนอทางเลือกหลายแบบพร้อมข้อดีข้อเสีย
- เขียนโค้ดตาม spec ที่กำหนด
- review และหาจุดที่อาจมีปัญหา
- debug ทีละขั้น
- validate ไอเดียของผู้พัฒนา

### สิ่งที่ผู้พัฒนาต้องทำเอง
- กำหนดทิศทางและตัดสินใจ
- เสนอไอเดียจาก domain knowledge
- ตั้งคำถามที่ดี เช่น "ทำไม?" "จำเป็นไหม?" "ใช้ทำอะไร?"
- ทดสอบจริงและสังเกตสิ่งผิดปกติ

### Prompt ที่ได้ผลดีในโปรเจคนี้

| รูปแบบ | ตัวอย่าง |
|--------|---------|
| ถามกว้างก่อน | "มีไอเดียอะไรบ้าง?" |
| เสนอไอเดียให้ validate | "ผมคิดว่า... คิดว่าไง?" |
| ถามเหตุผล | "ทำไมต้องเป็นสูตรนี้?" |
| ตั้งคำถามเชิงตรวจสอบ | "ทำไมโค้ดรันได้ทั้งที่..." |
| ขอทำทีละขั้น | "ถามก่อน / ทำเลย / ทีละไฟล์" |
| ปรับขอบเขต | "ไม่เป็นไร / เอาแบบง่ายๆ" |
